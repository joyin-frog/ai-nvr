# 已命名儿童进入 ROI 通知设计

## 目标

在现有 AI NVR 项目中新增一项能力：对已经在 TrackGallery 中手动命名的特定儿童，当其进入绑定的 ROI 区域时，立即通过现有通知渠道发送提醒。

第一版只支持“特定儿童”，不支持“任意儿童”。系统不新增年龄识别或人脸识别模型，继续复用现有的目标跟踪、命名和跨摄像头外观匹配链路。

## 范围

### 包含

- 在 ROI 配置中为区域启用“人物通知”
- 为单个 ROI 选择一个或多个已命名人物
- 为单个 ROI 设置冷却时间，默认 300 秒
- 当已命名人物进入该 ROI 时触发专用命中事件
- 复用现有 Webhook、钉钉、邮件通知渠道
- 冷却策略按“人物 + 摄像头 + 区域”去重

### 不包含

- 人脸识别
- 年龄识别或“儿童”自动判断
- 任意儿童检测
- 新的人物档案系统或全局 personId
- 在 ROI 中直接输入自由文本人物名
- 复杂规则编排页面

## 用户流程

1. 用户先在 TrackGallery 中给目标儿童命名，例如“小明”
2. 用户打开某个摄像头的 ROI 编辑器
3. 用户在某个 ROI 上启用“人物通知”
4. 用户从已命名目标列表中选择“小明”
5. 用户保存 ROI 配置
6. 后续当事件链路检测到“小明”进入该 ROI 时，系统立即通知
7. 同一人物在同一摄像头同一区域内，冷却期内不重复推送
8. 同一人物进入同摄像头的其他区域，或进入其他摄像头的绑定区域，仍然推送

## 现有能力复用

### 识别链路

第一版不改现有“识别”核心，继续依赖以下链路：

- 检测与跟踪生成 `trackId`
- 用户在 TrackGallery 中为 track 设置 `customName`
- 系统通过现有外观匹配能力为后续相似目标复用名称
- `track:enter-zone` 事件中携带 `trackName`

当前方案本质上依赖“已命名目标的再识别/名称传播”，不是新增人脸识别。

### 通知链路

继续复用现有通知出口：

- Webhook
- 钉钉机器人
- 邮件

## 数据模型

### ROI 扩展字段

在现有 `roi` 表基础上新增字段：

- `watch_list`：JSON 字符串，存储已命名人物名称数组，例如 `["小明", "小红"]`
- `notify_enabled`：布尔值，表示是否启用该 ROI 的人物通知
- `cooldown_seconds`：整数，表示该 ROI 的通知冷却时间，默认 300

### 字段语义

- `watch_list` 中保存的是 `customName` 字符串，而不是新的全局人物 ID
- 第一版默认同一部署中的关注人物名称应具有业务唯一性
- 若后续引入独立人物档案或人脸模板，再将其迁移为稳定 ID

## 前端设计

### ROI 编辑器

在 `RoiEditor` 中，为每个 ROI 条目增加“人物通知”配置区，包含：

- 开关：启用/关闭人物通知
- 多选列表：从已命名目标中选择一个或多个人物
- 冷却时间输入：默认 300 秒
- 说明文案：按“人物 + 摄像头 + 区域”去重

### 人物候选来源

第一版直接复用 `/api/tracks` 返回的数据，在前端提取所有 `customName` 并去重后作为候选列表。

约束：

- 只允许选择已有命名人物
- 不支持手工输入自由文本
- 若没有任何已命名人物，则该配置区展示为空状态提示

### 保存行为

ROI 新建或更新时，前端通过现有 ROI API 一并提交新增字段。

## 后端 API 设计

基于现有 ROI API 扩展字段，不改变接口风格：

- `GET /api/roi/:cameraId`
- `GET /api/roi`
- `POST /api/roi`
- `PATCH /api/roi/item/:id`

### 返回字段

每个 ROI 返回时增加：

- `watchList: string[]`
- `notifyEnabled: boolean`
- `cooldownSeconds: number`

### 写入字段

创建和更新 ROI 时支持接收：

- `watchList?: string[]`
- `notifyEnabled?: boolean`
- `cooldownSeconds?: number`

### 默认值

- `watchList` 默认空数组
- `notifyEnabled` 默认 `false`
- `cooldownSeconds` 默认 `300`

## 事件流设计

### 输入事件

继续监听现有 `track:enter-zone` 事件。该事件已经提供：

- `cameraId`
- `timestamp`
- `trackId`
- `label`
- `trackName`
- `zoneId`
- `zoneName`

### 命中逻辑

新增一个独立的后端命中服务，职责是监听 `track:enter-zone` 并执行以下流程：

1. 根据 `zoneId` 读取 ROI 配置
2. 若 `notifyEnabled = false`，跳过
3. 若 `trackName` 为空，跳过
4. 若 `trackName` 不在 `watchList` 中，跳过
5. 按“人物 + 摄像头 + 区域”检查冷却
6. 若未命中冷却，发出专用事件并执行通知

### 专用事件

新增事件：`track:watched-person-enter-zone`

建议载荷：

- `cameraId: string`
- `timestamp: number`
- `trackId: number`
- `label: string`
- `trackName: string`
- `zoneId: number`
- `zoneName: string`
- `cooldownSeconds: number`
- `sourceEvent: "track:enter-zone"`

这个事件语义上表示“某个被关注的人物进入了某个被配置的人物通知 ROI”，便于后续独立配置通知模板、事件展示和排查。

## 冷却设计

### 冷却键

冷却按“人物 + 摄像头 + 区域”去重。

推荐键格式：

`watch:{trackName}:{cameraId}:{zoneId}`

### 行为定义

- 小明进入 A 摄像头的前台区域后，5 分钟内再次进入 A/前台：不推送
- 5 分钟内进入 A 摄像头的仓库区域：推送
- 5 分钟内进入 B 摄像头的任意绑定区域：推送

## 命名变更同步

第一版必须处理“ROI 配置中的名字与 TrackGallery 改名脱节”的问题。

### 规则

当某个 `customName` 被修改时，需要同步更新所有 ROI 的 `watch_list`：

- 旧名 → 新名
- 若清空名称，则从 `watch_list` 中移除该名称
- 若更新后出现重复名称，去重保存

### 原因

第一版人物标识使用的是名字字符串。如果不做同步，ROI 配置会保留失效旧名，导致进入事件不再命中。

## 通知设计

### 渠道

专用命中事件继续复用现有通知执行器和渠道。

### 文案建议

通知文案应突出：

- 人物名
- 摄像头名或摄像头 ID
- 区域名
- 触发时间

例如：

- “儿童小明进入前台区域”
- “小明进入 Camera-B 的仓库区域”

### 调试信息

为便于排查，应在事件或告警 detail 中记录至少以下字段：

- `trackName`
- `zoneName`
- `cameraId`
- `cooldownKey`
- `cooldownSeconds`

## 错误处理与边界

### 未命名目标

未命名目标不触发本功能。这是第一版的有意限制。

### 跨摄像头匹配失败

若某儿童在其他摄像头中未被现有匹配链路成功关联到对应名称，则不会触发该摄像头下的 ROI 人物通知。

### 同名不同人

第一版默认关注人物名称具有业务唯一性。如果两个不同儿童使用相同名称，系统会把他们视为同一关注对象。

### ROI 禁用或删除

- 禁用 ROI 后，不再触发该 ROI 的人物通知
- 删除 ROI 后，其 `watch_list` 配置随 ROI 一并删除

## 测试策略

### 存储与 API

- ROI 新字段迁移成功
- ROI 列表接口正确返回新字段
- 新增、更新 ROI 时新字段可正确写入和读取
- 默认值正确生效

### 命中逻辑

- 命名人物进入已绑定 ROI，触发专用事件
- 未命名人物进入 ROI，不触发
- 命名人物进入未绑定 ROI，不触发
- 同人物同摄像头同区域冷却内重复进入，不触发
- 同人物同摄像头不同区域，触发
- 同人物不同摄像头，触发

### 前端交互

- ROI 列表能显示人物通知配置
- 只展示已命名人物候选
- 保存后刷新页面配置不丢失
- 改名后 ROI 绑定显示同步更新

## 实现边界

第一版仅实现以下最小闭环：

- ROI 中选择已命名儿童
- 进入指定 ROI 时触发命中事件
- 按“人物 + 摄像头 + 区域”冷却
- 通过现有通知渠道发送提醒

不在本轮实现中加入新的 AI 模型、自动儿童识别或人脸识别。

## 后续演进

若现有外观匹配在儿童场景下准确率不足，下一阶段可在不改 ROI 配置方式的前提下升级底层身份识别能力，例如：

- 人脸识别
- 儿童专用外观特征增强
- 独立人物档案与稳定 personId
- 上传参考照片建档

这些增强应继续复用本设计中的 ROI 配置、命中事件和通知链路，而不是推倒重做。
